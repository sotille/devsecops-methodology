# Contributing to the Techstream DevSecOps Methodology

Thank you for your interest in contributing. This repository provides organizational transformation guidance for DevSecOps programs: change management, team structure, governance design, security champion programs, and the phased playbook for moving from current state to a mature DevSecOps operating model. Contributions grounded in real-world transformation experience are especially valuable here.

## Code of Conduct

All contributors are expected to engage professionally and constructively. Contributions that are dismissive, personal, or unprofessional will not be reviewed.

---

## What We Welcome

- **Anti-pattern documentation** — real-world transformation failure patterns, with root causes and mitigations. These are among the most practically useful contributions to this framework.
- **Security champion program patterns** — role definitions, community of practice structures, training curriculum designs, and champion lifecycle management.
- **Organizational change management techniques** — approaches to overcoming specific resistance patterns, executive alignment strategies, and team adoption sequencing for different organizational contexts.
- **RACI and governance model variations** — adapted RACI matrices for different organizational structures (centralized security team, distributed platform team, federated model).
- **Industry-specific adaptation guidance** — how the transformation methodology applies differently in regulated industries, enterprise contexts, startup environments, or specific geographies.
- **Metrics and measurement improvements** — new KPIs for transformation progress, DevSecOps program health, and return on investment that are measurable and meaningful.
- **Transformation risk register additions** — common program risks with probability/impact assessments and mitigation strategies.

---

## What We Do Not Accept

- **Specific tool implementation guidance** — toolchain implementation belongs in the technical Techstream repositories (secure-ci-cd-reference-architecture, secure-pipeline-templates). This repository focuses on organizational and process methodology.
- **Fabricated case studies** — examples must be generic and representative, not attributed to specific organizations. Do not invent named clients, specific metrics, or fabricated outcomes.
- **Prescriptive organizational mandates** — the methodology provides patterns and guidance, not one-size-fits-all mandates. Contributions that remove organizational flexibility without strong justification will not be accepted.
- Vendor promotional content.
- Major structural changes without prior issue discussion.

## How to Contribute

### Reporting Issues

Use GitHub Issues to report: transformation pattern guidance that conflicts with common real-world experience, RACI model gaps for common organizational structures, missing coverage of important transformation failure modes, or outdated metrics references.

### Submitting Pull Requests

1. Fork and branch from `main` with a descriptive name.
2. For anti-pattern additions, use the established structure: Pattern Name, Root Cause, Manifestations, Mitigation.
3. Keep examples generic — representative of a class of organizational situation, not specific to any single organization.
4. Open a pull request with a description of what is being added or changed and the organizational context the contribution addresses.

## Documentation Standards

- Professional tone for transformation program leads, CISOs, engineering VPs, and organizational change managers.
- Use concrete, specific language. Avoid vague statements like "improve security culture" without defining observable indicators.
- RACI tables: Role, Responsible, Accountable, Consulted, Informed — consistent structure.
- Mermaid diagrams for transformation phase timelines, governance flows, and organizational model diagrams.
- ATX headers, bulleted lists for role responsibilities, numbered lists for sequential processes.

---

## Review Process

Pull requests are reviewed for grounding in real-world practice, scope alignment, documentation standards, and avoidance of prescriptive overreach. Initial responses within 5 business days.
